Fine-tuner BERT sur du droit : 100 % d'accuracy, et pourtant tout était faux

2026-07-02 · 14 min · 4 erreurs, 1 mensonge statistique, 1 leçon
Pipeline
Le projet 1 · Les données 2 · Piège : NameError 3 · Fine-tuning BERT 4 · Le mensonge du 100 % 5 · Les deux crashs Ce que je retiens

Je voulais un moteur de recherche juridique intelligent : classer des textes de loi avec BERT, générer des résumés avec BLOOM, indexer le tout dans Elasticsearch. Voici ce qui s'est vraiment passé, cellule par cellule.

D'abord, la carte du pipeline — vert : ça passe, jaune : ça passe mais méfiance, rouge : crash. Clique sur une étape pour y sauter :

setupColab + Drive merge2 datasets cleanregex + dédup trainBERT multiling. eval100 % ?! indexElasticsearch genBLOOM 560m

Regarde la colonne support : 18 exemples « Codes » contre 2808 « Autres ». Avec un déséquilibre pareil, un modèle qui répondrait « Autres » les yeux fermés aurait déjà 99,4 % d'accuracy. Le 100 % ne prouve pas que le modèle comprend le droit — il prouve que la tâche, telle que posée, est triviale.

// ce que j'aurais dû voir avant d'entraîner Vérifier data['Category'].value_counts() AVANT le split. Corriger par sous-échantillonnage, sur-échantillonnage, pondération des classes — ou repenser les labels : « Codes vs Autres » n'était peut-être pas la bonne question à poser à ces données.

5 · Les deux crashs de fin de course

Elasticsearch d'abord. Indexer les documents… sur localhost:9200. Sauf qu'un notebook Colab est une VM éphémère : il n'y a pas d'Elasticsearch qui écoute en local.

erreur ConnectionError : Failed to establish a new connection [Errno 111]

Puis BLOOM. Le fine-tuning génératif réutilise train_texts… qui est devenu un numpy.ndarray après le train_test_split. Et numpy n'a pas de méthode .apply — c'était du pandas dans ma tête :

train_texts_bloom = train_texts.apply(lambda x: f"Texte: {x}").tolist()
erreur AttributeError : 'numpy.ndarray' object has no attribute 'apply'
AttributeError Traceback (most recent call last) <ipython-input-45> in <cell line: 8>() ----> 8 train_texts_bloom = train_texts.apply(lambda x: f"Texte: {x}") AttributeError: 'numpy.ndarray' object has no attribute 'apply'
// les fixes Elasticsearch : un service managé (Elastic Cloud, Bonsai) ou l'installer dans la VM Colab le temps de la session. BLOOM : [f"Texte: {x}" for x in train_texts] — une list-comprehension ne demande pas la permission de pandas.

Ce que je retiens

① Regarder la distribution des classes avant de célébrer une métrique. ② Un notebook qui affiche 54 cellules propres peut cacher un état de kernel chaotique — l'ordre du fichier n'est pas l'ordre d'exécution. ③ Colab n'est pas ta machine : pas de localhost, pas de persistance. ④ .values transforme silencieusement ton pandas en numpy — et les erreurs n'apparaissent que 10 cellules plus loin.
