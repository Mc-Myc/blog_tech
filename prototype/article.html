Trois heures sur un healthcheck Docker qui mentait

2026-06-21 · 7 min de lecture · par Marcel
Sommaire
Le symptôme Ce que Claude a tenté Le coupable La leçon

Le conteneur était healthy. L'application était morte. Pendant trois heures, ces deux phrases ont été vraies en même temps.

Le symptôme

Tout commence par un déploiement banal. docker compose up -d, les services passent au vert, et pourtant le front renvoie des 502. Je lance une session Claude Code et je lui décris le problème.

● docker-compose.yml ⧉ copier
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 30s
  retries: 3  # ← tout a l'air normal, non ?

Ce que Claude a tenté

Claude a d'abord fait ce que j'aurais dû faire : vérifier que le healthcheck teste bien ce qu'on croit qu'il teste. Spoiler : non. L'image de prod était basée sur python:slim — et curl n'y est pas installé.

le piège — quand la commande d'un healthcheck n'existe pas, certaines configs le font passer pour réussi au lieu d'échouer. Le mensonge parfait.

Le coupable

Un healthcheck qui ne peut pas s'exécuter n'est pas un healthcheck qui échoue. La solution tenait en une ligne : remplacer curl par un outil réellement présent dans l'image.

● fix ⧉ copier
test: ["CMD", "python", "-c",
  "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"]

La leçon

Un test qui ne peut pas échouer n'est pas un test. C'est vrai pour les healthchecks, c'est vrai pour les tests unitaires, c'est la base du TDD que Claude me rappelle à chaque session : voir le test échouer d'abord.
