Un verdict de fraude ne se met pas en cache : autopsie d'un service résilient

Les trois patterns, à la lecture du code

Ce fichier de 60 lignes est une petite leçon d'architecture. Chaque pattern a sa forme signature sur ce blog — les voici détectés :

3 patterns dans 60 lignes !
circuit breaker

l'interrupteur : trop d'échecs → le circuit s'ouvre, on ne martèle plus un service mort

template method

le pochoir : la classe mère fixe le squelette, l'enfant ne remplit que doCall() et fallback()

deny-by-default

le bouclier : dans le doute, on bloque — l'échec du système ne doit jamais ouvrir la porte

src PaymentFraudAIService.java ⧉ copier
public class PaymentFraudAIService
        extends AbstractPythonServiceClient<Req, Res> ▤ template method

    @Override
    @CircuitBreaker(name = SERVICE, fallback = "callFallback") ⌁ circuit breaker
    @TimeLimiter(name = SERVICE)
    public Mono<FraudCheckResponse> call(String userId, FraudCheckRequest request) {
        return super.call(userId, request);
    }

    @Override
    protected Mono<Res> doCall(Req request) { ▤ le "trou" du pochoir
        return webClient.post()
                .uri("/fraud-detection")
                .bodyValue(request)
                .retrieve()
                .bodyToMono(FraudCheckResponse.class);
    }

    @Override
    protected FraudCheckResponse fallback(Throwable t) {
        log.warn("Fallback payment-fraud (DENY-by-default): {}", t.getMessage());
        return FraudCheckResponse.denyFallback(null); 🛡 deny-by-default
    }

Remarque la superposition : le template method structure (la mère gère timeout, logs, cycle de vie ; l'enfant fournit l'URI et le fallback), le circuit breaker protège (resilience4j ouvre le circuit après N échecs), et le deny-by-default décide de la politique quand tout a échoué.

La décision qui compte : NO-CACHE (ADR-011)

un verdict de
fraude ne se
cache pas !

La javadoc porte la vraie décision d'architecture :

ADR-011 — impossible de cacher un verdict de fraude : chaque transaction doit être évaluée individuellement. Le cache, réflexe de performance partout ailleurs, devient ici une faille de sécurité.
// pourquoi c'est la bonne politique Un fraudeur teste d'abord une petite transaction propre, puis frappe. Un verdict caché « OK » rejouerait la confiance du passé sur la transaction suivante. Symétriquement, le fallback DENY assume un coût : bloquer de vrais clients quand le ML est down — un arbitrage business, documenté, pas un accident.

La leçon

Les patterns ne sont pas de la décoration d'entretien d'embauche : dans ces 60 lignes, chacun répond à un risque précis — la panne en cascade (circuit breaker), la duplication entre 10 clients Python (template method), et la porte laissée ouverte (deny-by-default). Le jour où tu hésites à mettre un cache « pour la perf », demande-toi ce que vaut une réponse d'hier.
